# QA Plan generator

This skill is the **human handoff point** in the autonomous pipeline. Invoke when all GitHub issues for a PRD have been merged via the implementer-reviewer loop, and you want to hand the work to the human for acceptance testing before declaring the PRD shipped.

The human reads your generated QA plan, runs through each test case manually, and marks pass/fail. If all pass, the PRD ships. If any fail, the implementer-reviewer cycle resumes for the failing items.

## When NOT to use this skill

- Mid-feature, while issues are still being implemented
- For mechanical refactors where there's no user-observable behavior to test
- When the PRD didn't define acceptance criteria — push back: ask the user to run grill-me with you on the missing criteria first

## Process

### 1. Read the closed PRD

Locate the PRD in `docs/prds/` (or wherever the project stores them — check the map in CLAUDE.md). If multiple PRDs are candidates, ask the user which one. Read it fully — especially the **User Stories**, **Implementation Decisions**, and **Out of Scope** sections.

### 2. Read the closed issues

For each issue referenced by the PRD (or each issue with the `ready-for-agent` triage label closed after the PRD opened):

```bash
gh issue list --state closed --label ready-for-agent --json number,title,body,closedAt
gh issue view <number>
```

Map each closed issue back to the PRD's user stories. If you find user stories in the PRD that have no corresponding closed issue → **FLAG as a coverage gap**; the PRD is not fully implemented yet.

### 3. Inspect the merged code

For the period between the PRD's creation and now:

```bash
gh pr list --state merged --base main --json number,title,mergedAt,body
```

For each PR, read the diff if it touched a user-observable layer. Skim implementation-only changes (test files, refactors).

### 4. Generate the QA plan

Use the template below. The plan is a Markdown checklist designed to be readable in GitHub UI; the human copy-pastes pass/fail marks back as a comment.

<qa-plan-template>

# QA Plan: [PRD title]

- **PRD:** [link to PRD issue or doc]
- **Date generated:** [YYYY-MM-DD]
- **Issues merged:** [list of issue numbers]
- **Generated by:** `qa-plan` skill

## How to use this plan

1. Run each test case below in order.
2. For each, mark `[x] PASS` or `[ ] FAIL` and add a one-line note (especially for failures).
3. When done, post the completed checklist as a comment on the PRD.
4. If ALL pass → PRD is shipped (close it).
5. If ANY fail → reopen the failed acceptance criteria as new issues; the implementer-reviewer cycle resumes.

### 5. Publish the QA plan

Post the QA plan as a comment on the PRD's GitHub issue:

```bash
gh issue comment <PRD-issue-number> --body-file <tempfile>
```

Then notify the user explicitly:

> "PRD `<name>` is ready for QA. Plan: `<URL>`. Please run through the test cases and report back."

## Quality criteria for a good QA plan

- **Concrete, not abstract.** "Open Claude Code, type `/grill-me`, confirm it asks one question at a time" — NOT "verify grill-me works."
- **One test per user story.** If a user story has multiple acceptance criteria, that's multiple tests.
- **Runnable in <5 minutes per test.** If a test would take longer, split it.
- **No code-level testing.** This is *acceptance* testing; assume unit tests in the PRs already passed (the reviewer enforced that). Test user-visible behavior.
- **Idempotent steps.** Running a test twice should produce the same result.

## Edge cases

### A. PRD has user stories without clear acceptance criteria
List them in the "Coverage gaps" section. Do NOT invent tests on the user's behalf. Recommend the user run `grill-me` with you on the missing criteria.

### B. No closed issues yet match the PRD
The PRD is premature. Tell the user: *"QA plan generation requires at least one merged issue for the PRD. None found. Has implementation started?"*

### C. Issues were closed without being merged (wontfix, duplicate)
Skip those. Note in the QA plan: *"N issues from this PRD were closed unmerged (wontfix/duplicate); their user stories are unaddressed."*

### D. The PRD was modified mid-implementation
List the PRD's revision history (from issue edit history if available) and flag this to the user. Acceptance tests should follow the LATEST PRD version, but the human should be aware of the drift.

## Output format for the calling agent

After publishing the comment, return the canonical **GENERATOR trailer** (per [ADR-0005](../../../decisions/0005-output-shape-and-slicing-methodology.md) D1c and the "Output-shape standard" section of CLAUDE.md), as a fenced code block at the end of the terminal report:

```
RESULT: SUCCESS | STOPPED | INVALID_INPUT
REASON: <one sentence — e.g., "ready-for-human-qa — all acceptance criteria covered" or "needs-more-implementation — 2 user stories lack matching closed issues">
ARTIFACTS: <URL of the posted QA plan comment>
COVERAGE_GAPS: <count of user stories without acceptance tests>
```

**Canonical mapping note (for future maintainers).** This trailer replaces the prior `QA_PLAN_URL / COVERAGE_GAPS / RECOMMENDATION` shape with semantic content preserved:

- `QA_PLAN_URL` → **`ARTIFACTS`** (the URL is the canonical "artifact this skill produced").
- `RECOMMENDATION` (`ready-for-human-qa` / `needs-more-implementation`) **collapses into `REASON`** — on `RESULT: SUCCESS`, the REASON value begins with the prior RECOMMENDATION token followed by a short justification (e.g., `ready-for-human-qa — all 11 acceptance criteria covered`). On `RESULT: STOPPED` (e.g., implementation incomplete) or `RESULT: INVALID_INPUT` (e.g., PRD has no acceptance criteria), REASON describes the halt cause directly.
- `COVERAGE_GAPS` is preserved as a **permitted per-agent extension** (appended after `ARTIFACTS`) — its semantic (how many user stories lack acceptance tests) is unique to `qa-plan` and `/ship` or future orchestrators may consume it; collapsing it into `ARTIFACTS` or `REASON` would lose that signal.
