## Best-practice recommendations

- **Treat each sub-agent as a specialized assistant with a bounded job — not a general-purpose helper.** The opening framing of the video is explicit: "Sub-agents are specialized assistants that Claude can delegate tasks to." Specialization is the design intent, not an optimization. `(00:00:03)` Validates this project's per-subagent narrow-scope pattern (`prd-critic` judges PRDs only; `slicer-critic` judges decompositions only; `implementer` ships PRs from slices only) — generalist subagents are an anti-pattern by Anthropic's own framing.

- **Give every sub-agent its own conversation context window with a custom system prompt YOU define.** "Each sub-agent runs in its own conversation contacts window with a custom system prompt that you define." The author-controlled system prompt is the lever that turns a generic Claude into a specialized assistant; without it the sub-agent has no specialization. `(00:00:12)` Validates this project's `.claude/agents/<name>.md` convention where the entire body file IS the custom system prompt — there is no separate config layer.

- **Design sub-agents so they return a summary to the main thread and discard all intermediate work.** "When finished, it returns a summary to the main thread while all the intermediate work stays isolated." The summary-only return contract is what keeps the main context clean. `(00:00:19)` This is the load-bearing contract behind the project's GENERATOR/CRITIC trailer shapes (per ADR-0005 D1) — the trailer IS the focused-summary return; bodies are the discarded intermediate work that consumers of the trailer should not need.

- **Use sub-agents to manage main-context-window usage — that is the primary advantage.** "One of the main advantages of sub-agents is that they help manage context window usage." The video positions context hygiene as the chief organizing benefit, ahead of parallelism or specialization-per-se. `(00:00:24)` Argues that this project should evaluate new subagent proposals primarily by "does this keep the main context lean?" — a subagent that returns verbose intermediate state is failing its primary purpose regardless of how well it specializes.

- **Pass sub-agents exactly two inputs: the custom system prompt (configuration) + a task description (per-invocation, written by the parent).** "The sub-agent receives two inputs, a custom system prompt from your configuration file, and a task description written by the parent agent based on what you ask for." This is the canonical invocation shape — author-time configuration plus run-time task. `(00:00:47)` Validates the project's `Agent` tool invocation pattern (subagent_type + prompt); arguments against bolting additional input channels (env vars, side-channel files) onto subagents — the two-input contract is the standard.

- **Let sub-agents work autonomously once invoked — do not micro-manage their tool use from the parent.** "The sub-agent then works autonomously. When it reads files, edits files, or uses tools, none of these will appear in the main conversation." Autonomy is the model; the parent gives the task and gets the summary back, nothing in between. `(00:00:56)` Validates this project's no-interactive-checkpoint subagent pattern (critics emit verdicts in one shot; `implementer` produces a PR without back-and-forth with the orchestrator) — autonomous-by-default is Anthropic's recommendation, not just an autopilot optimization.

- **Accept the tradeoff: sub-agent invocations discard the conversation entirely after the summary returns.** "The entire sub-agent conversation then gets completely discarded." This is by design — you trade visibility into the journey for a clean main context. `(00:01:09)` Implication for this project: subagents must self-document inside their body (system prompt) and emit any reasoning the consumer needs IN the trailer/summary, because there will be no scrollback to inspect afterward. Supports the project's "self-contained subagent body" practice.

- **Be aware of the visibility cost: the main window loses insight into HOW the sub-agent reached its conclusions.** "The main window loses visibility into how the sub-agent reaches its conclusions and what it discovered along the way." This is named explicitly as a real cost, not glossed over. `(00:01:52)` Operational implication for this project: when a subagent's verdict is non-obvious (e.g., a critic BLOCK on a subtle rule), the verdict body should over-explain rather than under-explain — the consumer cannot re-derive the reasoning from a transcript that no longer exists.

- **Use the built-in sub-agents (general-purpose, explore, plan) before authoring custom ones — and add custom system prompts plus tool access when the built-ins don't fit.** "Claude Code includes several built-in sub-agents that you can use immediately... And you can also create your own sub-agents with custom system prompts and tool access." Custom subagents are the escape hatch for capabilities the built-ins don't offer, not the default tool. `(00:02:01)` Mild tension with this project's current pattern: the project has 8 author-defined subagents (6 critics + `slicer` + `implementer`) and uses NO built-ins explicitly. This is justified by the project's specific gating/critic needs but is worth a Phase 3 audit candidate (per ADR-0019 D8) — could any current subagent be replaced by a built-in plus a system-prompt-only override?
