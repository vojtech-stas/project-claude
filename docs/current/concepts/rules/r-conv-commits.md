# R-CONV-COMMITS

**R-CONV-COMMITS** is rule 4 in the [`reviewer`](../../../.claude/agents/reviewer.md) rubric. It hard-blocks any PR whose commit messages violate the project's tightened Conventional Commits format. The rule mechanically enforces CLAUDE.md cross-cutting rule #5 ("Conventional Commits, tightened") at PR review time and complements the `.githooks/commit-msg` server-side hook (PR #202) that catches violations before they reach the remote.

## What

Every commit on the PR's branch must follow `<type>(<optional scope>): <subject>` where type is one of the 10 allowed types:

- `feat`, `fix`, `docs`, `chore`, `refactor`, `test`, `perf`, `style`, `build`, `ci`.

Tightening rules from CLAUDE.md:

- **Lowercase subject** after the colon (`feat: add ship skill`, not `feat: Add Ship Skill`).
- **≤72 character hard cap** on the subject line.
- `Closes #<n>` belongs in the PR body, NOT the commit subject (separately enforced by [R-CLOSES](r-closes.md)).
- `Co-authored-by:` trailer required on every agent-authored commit.

## Why

R-CONV-COMMITS exists because **`git log` is the project's changelog** (CLAUDE.md cross-cutting rule #6). A consistent commit format makes the log skimmable, machine-parseable for changelogs and audit trails, and grep-friendly for `git log --grep '^feat'`. The lowercase + 72-char tightening optimizes for terminal-width display and downstream tooling consistency.

The type allowlist exists to prevent the long tail of one-off types (`wip`, `temp`, `fixup`) that defeat the changelog signal. Each allowed type maps to a known change-shape that downstream automation (release-notes generators, semantic-version bumpers) can act on.

## How to check

Inspect each commit message via `gh pr view`. Any non-conforming commit → BLOCK with `Conventional Commits: <commit-sha> subject "<subject>" violates <rule>`.

```bash
gh pr view <PR> --json commits --jq '.commits[].messageHeadline'
```

Per-commit checks:

1. Subject matches regex `^(feat|fix|docs|chore|refactor|test|perf|style|build|ci)(\([a-z0-9-]+\))?: [a-z].+$`.
2. Subject length ≤ 72 chars.
3. Body (if present) explains WHY, not what.

## Exemptions

- **Squash-merge commits on `main`** (post-merge): the reviewer never sees these since the merge itself runs after APPROVE.
- **Revert commits** auto-generated by `git revert` follow `Revert "..."` shape — accepted as conforming.

## Examples

- **`Feat: add ship skill`**: BLOCK — capitalized subject.
- **`feat: add ship skill that orchestrates the whole pipeline including the implementer subagent and the reviewer subagent and auto-merge`**: BLOCK — exceeds 72 chars.
- **`feat(kb): add 12 reviewer-rubric atomic notes + glossary back-edges`** (66 chars): PASS.
- **`wip: stuff`**: BLOCK — `wip` is not in the allowed type set.
