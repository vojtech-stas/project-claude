# Judge rationales + grafts

## winner pick: Trail of Record — artifact-first closed-loop observability
Judged on closing-the-loop fidelity: the user's purpose is (a) declared==measured genuinely evaluable and (b) genuinely watching agents work — in the environment where the user actually works, which per C2 is marathon resumed sessions where hooks never register.

Trail of Record wins because it is the only design where BOTH halves of the purpose function in that dominant mode. Its measured side is built exclusively on the layer the forensics proved indestructible (38/40 PRs verdict-parseable, 0.8s/PRD GraphQL, survived the synthetic-data era, already caught the real PR #650 unreviewed merge), so no dead hook or fixture can ever invalidate the comparison again — fixture quarantine is architectural, not procedural (C5 at its strongest). Its comparison semantics are the most complete: stable E-* edge ids abolish the string-pair matching that caused the original category error; `not-reached` and `not-exercised` states mean an in-flight run or an unexercised BLOCK loop never renders as fake drift — the exact symptom ('totally different results') being fixed. Its Live tab Lane A (artifact run-progress) is the only live surface of the three that works during a marathon /ship run, while Lane B + honest status pills preserve real-time texture when capture is alive. Verifiability is maximal: day-1 retro-proof against real closed PRD #640 (real datum end-to-end with zero dependence on new capture), PR #650 must-flag as a slice proof, and the final PRD shipping itself through the pipeline it measures. Its honest weaknesses — outputs-as-proxies for dispatches, verdict-posting as prose convention — fail LOUD (red missing edges), which is the correct failure direction for this project.

PULSE loses on the evidence: it re-centers the system on the layer proven to die exactly when the user works. Its 34/36 runtime-coverage headline stacks three unverified assumptions (CRITIC trailer present in tool_response tail — the forensics observed tool_response as {status, prompt}, possibly containing no agent output at all; subagent tool calls firing parent-session hooks; PostToolUse(Skill) reliably popping the per-session stack). Because runtime events feed its comparison, a missed stack pop or misattribution produces false 'unexpected' reds — reintroducing the precise fake-drift class this refactor exists to kill. Its golden run requires a fresh session, recreating the P5 verification-environment-infidelity pattern (verifying in an environment that differs from the user's in exactly the dimension under test). The SSE rebuild, js/* split + static route, vendoring, per-session state machine, and dual-provenance reconciliation engine make it the heaviest build for a single-dev local tool (C7 strain), with the highest migration surface.

Two-Plane is a close second (43 vs 44) — the same sound artifact-first inversion, leanest staging, safest migration. It loses on the primary lens by two concrete seams: (1) no 'not-reached' state, so its default both-view (most recent PRD = the in-flight one) paints required-but-not-yet-reached edges as red 'missing' and the scorecard FAIL during every active run — declared==measured is only well-defined post-hoc, and the user watching mid-run sees fake drift again; (2) in dead-hook sessions its Live tab degrades to a banner pointing at another tab, with no artifact run-progress feed — the 'see the agents work' half goes unserved in the dominant working mode, where Trail of Record's Lane A serves it. Lesser dings: edges keyed by name pairs rather than stable ids (the historically defect-prone mechanism), codebase-critic and side workflows left dark, and fantasy appetites (~1 focused day for PRD 1's four substantial slices, against a repo history of 34% fix-churn and a 4-fix-PR topology saga).

## winner pick: Trail of Record — artifact-first closed-loop observability
ROBUSTNESS LENS VERDICT. The decisive fact from the forensics is that the failure mode is not 'hooks are buggy' but 'hooks never register in resumed sessions' (process-retrospective: frozen 2026-06-05 14:16 to 06-10 because the orchestrator's long-lived session never registered hook config — and that IS the user's working mode). Any design whose primary measurement lives on runtime events is betting against the empirically dominant session shape.

PULSE (32) makes exactly that bet. Its mitigations are real (durable gh floor, beacon-before-parse, rejects file, in-hook python3 self-test — the best capture engineering of the three), but the architecture's center of gravity is wrong: (a) the headline ~34/36 live edge coverage exists only in fresh sessions, so its golden-run PASS is evaluable only when capture is alive — durable measurement survives dead hooks only as a degraded artifact-spine floor, brushing the C2 line; (b) runtime events participate in declared==measured PASS, so fixture pollution CAN reach the verification verdict — its quarantine is detector-backed (writer-stamp, events-vs-beacon counts), not structural, the weakest C5 of the three; (c) the per-session skill-stack state files have no lifecycle story (missed PostToolUse(Skill) pops, crashed skills, stale state → silent misattribution); (d) SSE on the stdlib server is less fatal than D1 claims (verified: ThreadingHTTPServer, so a held connection doesn't block the handler) but is still stream-state code that breaks across the routine post-merge restarts (#623 class) for a latency gain nobody needs at minute-scale stages; (e) the gh-touchpoint bash-regex parser is brittle to command-shape drift (its own risk register) and duplicates what the collector recovers authoritatively. It is also the heaviest build (correlation stacks + SSE + gh parser + mismatch reconciliation + js split + static route + vendoring) for a single-developer local tool.

Trail of Record (42) and Two-Plane (41) share the correct core bet: GitHub artifacts are the only measured side that cannot be killed by a dead hook or polluted by a fixture — the deepest possible C5 wall (measurement never reads the log) and full C2 compliance. They differ at the margins, and the margins favor Trail of Record on this lens. (1) Lane A — artifact-fed run progress polled at 20-30s — means the user's 'see the agents work' function itself SURVIVES dead hooks in a marathon session; Two-Plane's Live tab is hook-fed only, so under the dominant session mode the watch surface degrades to a banner pointing at a verification scorecard. (2) Trail of Record's comparison semantics include 'not-reached' for in-flight runs; Two-Plane's state table lacks it, so an open mid-run PRD can paint downstream required edges as red 'missing' — false reds are precisely the trust-killer this whole saga was about (the structurally-guaranteed-red both-mode). (3) Byte-cursor incremental polling beats Two-Plane's timestamp-based after= cursor. (4) Stable edge ids (E-*) make the comparison report, UI, CLI, and golden-run proof key on one contract, abolishing the string-pair matching class of defects forever. Two-Plane wins YAGNI (mostly deletion, 4 PRDs, server stays one file) and migration safety (CHECK 9 names literally untouched), and its restraint is genuinely admirable — but it under-serves half the user's verbatim purpose in the session mode that actually matters.

What the winner glosses over (flagged adversarially): its SSE-deletion justification (a) is factually off — the server is ThreadingHTTPServer, so 'occupies the handler' is wrong; the valid arguments are restart fragility and C2 honesty. Its staging header says 'six PRDs' but lists five. It specifies error beacons but not raw-stdin preservation on parse failure. Its fixture routing is pattern-regex only (no sandbox env override). Byte-cursor behavior under log truncation/rotation (which already happened once, 06-10) is unstated. The posting convention is prose-enforced — all three designs share this risk, and all three correctly make non-compliance loud red rather than silent. None of these is structural; all are graftable from the losers.

The clincher: Trail of Record's PRD 1 slice 1 proves the load-bearing layer first (real closed PRD #640's trail flowing spec→collector→comparison→pixel, zero hook dependency, works in the user's current marathon session on day 1), whereas PULSE's slice 1 proves the layer that should never have been load-bearing.

## winner pick: Two-Plane Closed Loop — GitHub-measured pipeline truth + script-hook live pulse (minimal surgery)
All three designs converge on the same evidence-forced conclusions (script-file hooks with python3 not jq; GitHub collector with closingIssuesReferences + retry-once-401; one evidence-annotated spec; fixture quarantine; server-side comparison; golden-run acceptance) — the real differentiators are (1) which layer carries declared==measured and (2) how much surface gets built. PULSE loses on both: it makes the layer that C2 proves dies exactly when the user works (marathon resumed sessions) the PRIMARY measurement source, stacks four unverified mechanics on top of it (trailer-in-tail despite the topology forensics observing only {status,prompt}; PostToolUse(Skill) stack push/pop where one missed pop corrupts every later dispatched_by stamp; gh-command regex coupled to skill-body CLI surface; subagent calls firing parent-session hooks), and spends single-dev budget on SSE-over-stdlib-http.server, a per-dispatch correlation system, a js/* split requiring a new static route, and CDN vendoring — while its own fallback paths converge on the artifact-first architecture the other two start from. Its mismatch state even makes golden-run PASS fragile: one dropped hook event manufactures a false FAIL on a healthy run. Trail of Record and Two-Plane share the correct artifact-first architecture; they differ in spend. Trail of Record costs 6 PRDs/~4 weeks, an 8-module server split the code-health audit itself sanctioned skipping (single-file reorder named as the acceptable fallback), two ADRs, and two new numbered CLAUDE.md rules where one rule plus critic-rubric lines suffices (parsimony per ADR-0046 D1). Two-Plane reaches near-identical capability in 4 PRDs/~2.5 weeks with one new ~300-line module, one ADR, one new rule, zero CHECK 9 migration risk (check functions never move; PIPELINE stays in server.py), and the discipline to delete rather than rescue (SSE, deriveMeasuredEdges, children[]) and to not invent correlation on a feed that dies in resumed sessions. Its PRD 2 slice 1 — the verdict-posting convention proving itself by flipping its own PRD's edge from missing to pass — is the best single slice across all three designs. Two-Plane's genuine gaps (no not-reached comparison state for in-flight PRDs; Live tab degrades to a banner in marathon sessions instead of an artifact-fed run lane; 700-char tail can clip long trailers; a non-credible ~1-day appetite for the 4-slice PRD 1) are all cheaply graftable from the losers without disturbing its economy — so the minimal-surgery design wins, amended.

# Assumption verdicts

## [REFUTED] Agents actually follow the verdict-posting convention: it is prose-enforced in /ship and /qa-plan bodies. If agents skip the comment, prd-critic/slicer-critic edges go dark again — the comparison shows red 'missing' (loud), coverage regresses to ~7/11 stages, and the user sees drift caused by discipline, not behavior.
The specific mechanism named — prose enforcement in /ship and /qa-plan bodies, executed by the main-agent orchestrator — has a measured compliance record in this repo's own GitHub trail of 0–17.5%, decaying to 0% within days. Four existing prose-enforced posting conventions, all measured via gh on 2026-06-10: (1) Pipeline-metadata footer (prd-critic VERDICT+ROUND appended to every PRD body, mandated in .claude/skills/to-prd/SKILL.md L31 since 2026-05-19 commit 8a9d0b5, AND prose-verified by /ship 'Stage 2.5 verification' L79): 7/40 closed PRDs comply (17.5%); the 7 cluster in a 2.5-day window (#512–#547, 06-02→06-04); 0/13 since 06-05 — including 11 PRDs (#562–#640) carrying the canonical 'PRD:' title proving they went through /to-prd; /ship's own stage-2.5 prose check was skipped 13 consecutive times. (2) QA-plan persisted as PRD comment (/qa-plan step 3, ADR-0020 D4): 1/32 closed PRDs (#474 only). (3) Proof-posting (/ship step 6 'chore(qa): add visual proof' commit + qa-proof PR comment, shipped 06-05 commits 5009cc0/e14bb0d): exactly 1 PRD ever (qa-proof/574, committed the same day the convention shipped, PR #578); 0 of the 8 subsequent PRDs; 1/10 recent PRs have any qa-proof comment. (4) needs-human-check residual issues (/qa-plan step 5): 0 issues have ever carried the label. The lone high-compliance counterexample proves the distinction is structural, not cultural: reviewer verdict comments measure 39/40 merged PRs (97.5%, ROUND parseable 38/40; re-verified independently) — but the reviewer is a freshly-dispatched SUBAGENT whose gh pr comment IS its primary output channel (reviewer.md L316), not a main-agent bookkeeping side-step; even it was bypassed once (PR #650, merged 06-10, zero verdict comments, no labels). prd-critic/slicer-critic run inside /to-prd//to-issues in main-agent context, so the design's posting step lands on the executor class with 3–17% measured follow-through. Compliance decay tracks session age — the same marathon-session degradation as constraint C2, at the discipline layer. Consequence for the design: red-'missing' would be the permanent steady state for prd-critic/slicer-critic edges (not a degraded exception), the golden-run acceptance could never PASS those edges, and ~7/11 coverage is the floor the system would live at, not regress to.
MITIGATION: Three layers, ordered by proven effectiveness: (a) Relocate the posting into the critic subagents' own output contracts — clone the reviewer pattern (97.5% measured): prd-critic.md and slicer-critic.md gain an authorized terminal action 'gh issue comment <PRD> --body-file <tempfile>' posting their own fenced CRITIC trailer to the PRD issue before returning; never rely on the orchestrator to relay. (Note: /to-prd runs prd-critic in main-agent skill context today — this requires the critic dispatch to be a true subagent with gh comment authorization, which it already is per the critic roster.) (b) Mechanical backstop in the only gate that empirically always fires: extend tools/ci-checks.sh (the R4 required-status-check 'ci' job, runs on every PR via GitHub Actions — cannot be skipped by a degraded session) with a check that any PR whose body has 'Closes #<slice>' resolves the parent PRD via the GraphQL subIssues field and FAILs unless the PRD carries a parseable 'VERDICT:'+'ROUND:' prd-critic comment (and slicer-critic comment once slices exist); the same check lints the trailer to a fixed key schema, killing PR #559-class ROUND drift. This converts the convention from prose to merge-blocking. (c) Second evidence source so edges survive discipline failure anyway: capture v2 stores the TAIL of agent_complete tool_response (per C4b — the CRITIC trailer lives there), and the collector's per-edge verdict predicate is satisfied by EITHER the GitHub comment OR the runtime tail event; the comparison layer renders 'evidence-missing (discipline)' as a distinct per-edge state from 'behavior drift', and the unreviewed-merge detector (PR #650 class: merged, no verdict comment, no trivial label) ships regardless, because even the 97.5% convention has bypasses.

## [HOLDS_WITH_CAVEAT] GitHub stays fast and the gh keyring flake stays benign: the design banks on 0.8s/PRD GraphQL and retry-once-on-401 sufficing. Sustained auth flapping, tightened rate limits, or large PRD counts would stale the authoritative lane that the whole architecture promotes to primary.
Tested live on this host (2026-06-10) against vojtech-stas/project-claude. (1) SPEED REPLICATES: the full batched per-PRD trail query (issue + comments + subIssues + sub-issue comments + timeline CrossReferencedEvents + PR closingIssuesReferences + PR comments) ran 5/5 times at 0.77-0.97s, 46,415 bytes, GraphQL cost=1 point — matching the forensics' 0.816s/46.6KB claim exactly. (2) SCALE SURVIVES BUT THE FORENSICS UNDER-SIZED IT 6x: `gh issue list --label prd --state all` returns 95 PRDs (all closed, 0 open), not the ~15 the trail-assessor's '<15s cold' estimate assumed. Measured full 95-PRD cold backfill: 73.2s wall, 2.2MB, ~95 GraphQL points, 0 failures. Post-test budget: core 4952/5000, GraphQL 4706/5000 — the heaviest realistic workload uses <2% of one hour's points, so rate limits would need a ~50x tightening to bite. Because closed PRDs are immutable and 100% of the corpus is closed, the cache-forever design reduces steady-state API load to open PRDs only (currently zero). (3) FLAKE DID NOT REPRODUCE: 0 auth failures in ~171 gh calls this session — 40-call sequential REST burst, 95 sequential GraphQL backfill, 8-way parallel REST, and 16-way parallel GraphQL specifically to stress Windows keyring retrieval contention (16/16 exit 0). Combined with the forensics' historical sample (3/20 transient 401s, 3/3 recovered on immediate retry), the observed failure class is transient-with-instant-recovery, not sustained flapping; no evidence of a sustained-flap mode exists. CAVEATS that block a clean HOLDS: (a) the cold path is 73s, not <15s — the immutable-closed-PRD cache is therefore load-bearing, not an optimization; an uncached implementation would stall every dashboard load ~73s and the lane genuinely would feel stale; (b) retry-once is unproven against sustained flapping because zero 401s occurred today — a 0-failure day cannot statistically bound a bursty failure mode that hit ~15% on one prior day; (c) retry-once is structurally useless against true auth death (revoked/expired gho_ token): every retry 401s identically, so the design needs to distinguish 'flake' from 'auth dead' rather than assume all 401s are transient.
MITIGATION: Four-layer degradation so the GitHub lane fails stale-and-honest instead of dark. L1 — cache-first serving (make the load-bearing cache explicit): persist closed-PRD trails forever keyed on (issue number, closedAt); on ANY fetch failure serve the cached trail stamped with fetched_at, and the dashboard renders a 'measured as of N min ago' staleness banner reusing the honest-banner pattern already shipped in commit 6ed055d (#648). Sustained flapping then stales only open-PRD freshness; the historical measured graph never disappears. L2 — replace retry-once with bounded backoff: 3 attempts at 0s/2s/8s treating 401/5xx/timeout uniformly; on budget exhaustion mark the lane DEGRADED in the dashboard health panel with the verbatim error (failure-loudness principle from the event-pipeline autopsy — never exit-0-silent). Three consecutive failed cycles = 'auth dead' state, surfaced as a distinct red badge telling the user to run `gh auth status`, instead of being misread as transient flake. L3 — keyring bypass escape hatch: gh prefers GH_TOKEN over the Windows keyring, and keyring retrieval is the only empirically observed auth-failure mechanism; if 401s persist across one backoff cycle, the collector sets GH_TOKEN for its gh subprocesses from a local gitignored token file (documented in bootstrap.sh), removing the flake mechanism entirely. L4 — offline floor via git log: squash-commit subjects carry the PR number ('(#644)'), Co-Authored-By trailers carry agent provenance, and commit dates carry merge times — a zero-API git-log reconstructor recovers merged/provenance edges so declared-vs-measured degrades to 'merge-verified, verdicts unknown' (edges marked stale-unverified) rather than going dark, even fully offline.
