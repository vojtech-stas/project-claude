## SUMMARY
# Event-Pipeline Autopsy — findings

## 1. The 13 registrations (F:\project_claude\.claude\settings.json)
**6 simple script calls** (SessionStart→session-start.sh, SessionStart→dashboard-autostart.sh, UserPromptSubmit→user-prompt-submit.sh, PreToolUse(Edit|MultiEdit|Write)→pre-tool-edit.sh, PreToolUse(Bash)→pre-tool-bash.sh, Stop→stop-reviewer-gate.sh) and **7 inline-bash-with-jq pipelines**, all of which pipe a jq-built JSON line into `.claude/hooks/log-event.sh`: PreToolUse(Agent)→agent_start, PreToolUse(Skill)→skill_invoke, PostToolUse(Edit…)→subagent-edits.log nudge (grep, no jq envelope), PostToolUse(Agent)→agent_complete, PostToolUse(Bash)→bash_complete, PostToolUse(AskUserQuestion)→grill_qa, Stop→session_stop. The Stop logger uniquely takes session_id from env `${CLAUDE_CODE_SESSION_ID}` instead of stdin.

## 2. Simulations: the commands themselves are CORRECT
All 5 mission-required simulations (PostToolUse Bash/Agent, PreToolUse Agent/Skill, Stop, plus user-prompt-submit.sh with a `/ship` prompt) run under bash with jq on PATH **append exactly the right events** (`bash_complete` with command, `agent_complete` with input/output, `skill_invoke` both sources, `session_stop`). With jq removed from PATH: **zero lines appended to BOTH files, exit 0, error only on suppressed stderr** (`jq: command not found`). Under cmd.exe the inline shape dies on single-quote parsing (jq syntax error). So the dead pipeline is environmental, not syntactic. (All test lines were removed; logs restored byte-exact to as-found: workflow-events.jsonl empty, hook-fires.jsonl 4 lines.)

## 3. Root cause of the dead capture (empirically proven via two live headless sessions)
I reproduced the 2026-06-10 anomaly with real `claude -p` sessions (v2.1.156) under `--debug` and read claude's own per-session debug logs (C:\Users\vojte\.claude\debug\169ad155….txt and 18634e5b….txt):

- **Hook executor**: confirmed from binary strings + debug log `Using bash path: "C:\Program Files\Git\bin\bash.exe"` — hooks spawn as `spawn(command, [], {shell: GIT_BASH})`, i.e. `bash.exe -c '<command>'` with the claude process env (the binary even contains: `Hook "…" requires bash but Git Bash was not found… add "shell": "powershell"`). So Git Bash IS the executor; the simple-script hooks run fine.
- **Smoking gun** (run 1, Stop hook output, verbatim): `/c/Users/vojte/bin/jq: /c/Users/vojte/bin/jq: cannot execute binary file`. Inside the claude-spawned hook bash, `jq` resolves to `C:\Users\vojte\bin\jq.exe` (seen as extensionless `/c/Users/vojte/bin/jq` via cygwin suffix-magic; byte-identical copy of the winget jq.exe; carries a Zone.Identifier/Mark-of-the-Web stream; `~/bin` precedes the winget jq dir in PATH) — and **exec fails with ENOEXEC in that context**. Failure is intermittent per invocation: in run 2, the PostToolUse(Bash) pipeline's two extraction `jq -r` calls failed (yielding `session_id:"" command:""`) while its final `jq -cn` envelope succeeded, and the Stop `jq -cn` failed again (2/2 sessions). The same alias/commands ran **0-failures-in-145** manual executions (bash-, PowerShell-, and node-`spawn({shell})`-launched), so the ENOEXEC is specific to the claude.exe-spawned hook environment (MotW/SmartScreen or msys env nuance — unresolved residual; the fix doesn't depend on it).
- **Why zero trace**: `log-event.sh` line 44 `[ -z "$LINE" ] && exit 0` exits on the empty pipe BEFORE its own beacon write (line 69); every extraction jq is `2>/dev/null`; every hook exits 0. Silent total data loss by construction.

**This explains the 2026-06-10 anomaly exactly**: session-start/dashboard-autostart/user-prompt-submit/stop-reviewer-gate beacons need only printf+date → they appear; the Stop logger's jq dies → empty stdin → log-event.sh exits before beacon → 0 workflow events, 0 log-event beacon. UserPromptSubmit logs only when the prompt starts with `/` (it didn't). The 2026-06-04 `{"session_id":"","command":""}` events are the partial variant (envelope jq survived, extractions died). On 2026-05-31 capture fully worked (archive shows real session a0aa9ea6 with full commands AND `session_stop` carrying the env-sourced sid — so `CLAUDE_CODE_SESSION_ID` IS exported to hooks).

## 4. Emitters vs consumers (mismatch map)
Emitted events: agent_start{ts,session_id,event,subagent_type,input}, agent_complete{+output}, bash_complete{ts,sid,event,command}, skill_invoke{ts,sid,event,skill,source:skill_tool|user_typed}, grill_qa{ts,sid,event,question,answer}, session_stop{ts,sid-from-env,event}; session-start.sh emits **no workflow event at all**; PostToolUse(Edit) writes plain text to subagent-edits.log via LOGFILE override. Consumers (F:\project_claude\dashboard\server.py): `/api/runs` (line 1314) buckets purely by `session_id` (`"" → "unknown"`) and orders by `ts`; `/api/events` (line 1458) is a raw SSE tail. Mismatches: (a) degraded empty-sid events from ALL sessions merge into one fake "unknown" run — this is what the Live tab actually displayed; (b) no session_start event → run boundaries inferred from first tool event, idle/early session invisible; (c) session_stop's env-sourced sid diverges from every other event's stdin-sourced sid; (d) `?session=` single-run fetch breaks at the first non-matching line after a match — interleaved concurrent sessions (orchestrator + worktree subagent) truncate; (e) log-event.sh's #572 "session_id stamping" is a tautological no-op (extracts sid from the line and merges it back into the same line — can never fill a missing sid).

## 5. Live tab + measured topology (dashboard/index.html)
Live tab **polls**: `renderLiveGraph()` fetches `/api/runs?n=10` metadata + `/api/runs?session=<id>` per selection, on a 30 s `setInterval` (line 4128). `EventSource` count in index.html: **0** — the `/api/events` SSE endpoint has no consumer (dead code), and it also has a Last-Event-ID resume bug (if no new lines exist, `start_line = line_index` resets to a stale value → full duplicate replay every second) plus full-file `readlines()` each second. Real-time updating requires wiring an EventSource client (or keeping polling) — but the actual blocker was always upstream: no events to show. The Architecture tab's "measured" topology (`fetchMeasuredEvents`, line 1351) derives edges from skill_invoke + agent_complete events of the last 20 sessions of workflow-events.jsonl, while "declared" parses settings.json/filesystem — with the log dead-or-synthetic, **declared ≠ measured is structurally guaranteed**. Until 2026-06-10 the log was 3648 lines dominated by synthetic fixtures (session `sess-test-001`; one event's command literally says "Recreate synthetic log with future timestamps so it's the latest session"), now archived in workflow-events.jsonl.synthetic-archive-20260610; the live file was truncated to 0 at 13:42.

## 6. Telemetry display defect
`discover_hooks`/`_read_hook_name` (server.py lines 382/493): the `.sh` regex matches `hooks/log-event.sh` INSIDE every inline command, so **all 7 inline registrations collapse to the single name "log-event" sharing one telemetry bucket**; the per-logger name branches ("agent_start logger" etc., keyed on `workflow-events.jsonl` appearing in the command) are dead code for the current settings.json. Also the log-event beacon counts successful deliveries, not attempts, so "fired N×" cannot reveal the jq failure class. Beacon split-brain: the 6 script hooks beacon to `$CLAUDE_PROJECT_DIR/.claude/logs` (worktree-local under I4a isolation) while log-event.sh git-common-dir-resolves to the main root; dashboard-autostart.sh likewise spawns `$CLAUDE_PROJECT_DIR/dashboard/server.py` whose `REPO_ROOT = Path(__file__).parent.parent`, so a worktree-first session serves the worktree's (empty) logs on the contested port 8765.

## KEY FACTS

- All inline hook commands are syntactically correct bash: all 5 simulated commands (PostToolUse Bash/Agent, PreToolUse Agent/Skill, Stop) appended correct events when run manually with jq on PATH; with jq absent they append zero lines to both logs, exit 0, error only on suppressed stderr
- Hook executor on Windows (claude v2.1.156) is Git Bash: spawn(command, [], {shell: 'C:\Program Files\Git\bin\bash.exe'}) — confirmed by binary strings ('Hook "…" requires bash but Git Bash was not found') and the session debug log line 'Using bash path: "C:\Program Files\Git\bin\bash.exe"'
- Root cause of dead capture, observed in two live headless sessions via C:\Users\vojte\.claude\debug\<session>.txt: inside claude-spawned hook bash, jq resolves to C:\Users\vojte\bin\jq.exe (extensionless alias /c/Users/vojte/bin/jq via cygwin suffix-magic; MotW-flagged; shadows the winget jq.exe in PATH) and exec fails ENOEXEC 'cannot execute binary file' — Stop logger failed 2/2 sessions; PostToolUse extraction jqs failed while the envelope jq -cn succeeded, producing {"session_id":"","command":""}
- The same jq alias and commands succeeded 145/145 times when spawned manually (bash, PowerShell, node spawn with shell option) — the ENOEXEC is specific to the claude.exe-spawned hook environment
- The failure is invisible by design: log-event.sh exits on empty stdin (line 44) BEFORE writing its own hook-fires beacon (line 69); all extraction jqs are 2>/dev/null; hooks always exit 0 — exactly matching the 2026-06-10 anomaly (4 script beacons, no log-event beacon, 0 events)
- CLAUDE_CODE_SESSION_ID IS exported to hook processes (2026-05-31 archive contains session_stop events carrying the real session id), and capture fully worked end-to-end on 2026-05-31; 2026-06-04 produced empty-sid/empty-command events (partial jq failure); by 2026-06-10 capture was fully dead
- workflow-events.jsonl as found: 0 lines (truncated 2026-06-10 13:42); the 3648-line synthetic-archive is dominated by fixture data (session sess-test-001, commands literally describing synthetic log creation) — the 'measured' topology and Live tab were rendering test fixtures, guaranteeing declared != measured
- /api/runs (dashboard/server.py:1314) buckets events purely by session_id with '' -> 'unknown' (merging all degraded events into one fake run) and the ?session= branch truncates interleaved concurrent sessions
- /api/events SSE endpoint (server.py:1458) has ZERO consumers (0 EventSource occurrences in dashboard/index.html) and a Last-Event-ID resume bug causing duplicate replay; the Live tab polls /api/runs every 30 s instead
- Dashboard hook telemetry collapses all 7 inline registrations to one shared 'log-event' name/bucket (server.py _read_hook_name line 382: the .sh regex matches hooks/log-event.sh inside inline commands; the per-logger name branches are dead code)

## DEFECTS
- [critical] Hook runtime environment: PATH-resolved jq (C:\Users\vojte\bin\jq.exe via extensionless cygwin alias) used by all 7 inline loggers in F:\project_claude\.claude\settings.json: In claude-spawned Git Bash hooks, exec of the jq alias fails ENOEXEC ('cannot execute binary file') intermittently-to-always (Stop logger 2/2 sessions, extraction jqs 2/2), killing every inline event logger. Same commands are 100% reliable in user-spawned shells, which is why manual QA ('log-event.sh works when invoked by hand') never caught it. This is the root cause of the dead capture pipeline.
- [critical] F:\project_claude\.claude\hooks\log-event.sh lines 43-44 vs 69; all inline commands' 2>/dev/null; exit 0 convention: Silent-failure design: empty stdin exits BEFORE the hook-fires beacon is written, so the #644 telemetry system is structurally blind to the very failure class (jq death) it was built to expose; no stderr, no non-zero exit, no beacon, no event.
- [critical] F:\project_claude\.claude\logs\workflow-events.jsonl (now archived as .synthetic-archive-20260610) + dashboard measured topology (index.html fetchMeasuredEvents line 1351): The event log was contaminated with 3648 lines dominated by synthetic fixture sessions written during development/QA; the 'measured' graph and Live tab rendered fixtures as if real, making declared-vs-measured comparison meaningless and masking that real capture was dead since ~2026-06-01.
- [major] PostToolUse(Bash)/Agent inline pipelines + dashboard/server.py _serve_runs line 1415 ('' -> 'unknown'): Partial jq failure writes schema-empty events ({"session_id":"","command":""}); the server merges all such events across every session into one 'unknown' pseudo-run, corrupting session bucketing, duration spans, and the Live tab graph.
- [major] F:\project_claude\dashboard\server.py _serve_sse (line 1458) + dashboard/index.html (0 EventSource usages): /api/events SSE endpoint is consumer-less dead code; it also re-reads the whole log every second and has a Last-Event-ID resume bug (start_line reset to stale line_index triggers full duplicate replay). Live tab 'real-time' is actually a 30 s poll of /api/runs.
- [major] F:\project_claude\dashboard\server.py _read_hook_name (line 382) / discover_hooks (line 493): All 7 inline registrations collapse to one shared 'log-event' telemetry name/bucket because the .sh regex matches the piped log-event.sh path; the inline-logger naming branches are unreachable for current settings.json; per-hook 'fired N× / never fired' badges cannot distinguish which logger fired, and count successful deliveries rather than attempts.
- [major] F:\project_claude\.claude\hooks\session-start.sh, user-prompt-submit.sh, pre-tool-*.sh, stop-reviewer-gate.sh, dashboard-autostart.sh (beacon paths) + dashboard-autostart.sh server spawn: Worktree split-brain: script hooks beacon to $CLAUDE_PROJECT_DIR/.claude/logs (worktree-local under I4a worktree isolation) while log-event.sh resolves to the main root via git-common-dir; dashboard-autostart launches the worktree's server.py whose REPO_ROOT is the worktree, so a worktree-first session binds port 8765 serving empty logs.
- [major] Event schema: session-start.sh (no event emitted) + Stop registration (sid from env) + server.py ?session= branch (line 1362 break-on-non-match): No session_start event exists, so run boundaries derive from the first tool event; session_stop sources sid from env while all other events use stdin payload (divergent contracts); the single-session fetch truncates when two sessions' events interleave in the shared log (real risk with concurrent orchestrator + subagent sessions).
- [minor] F:\project_claude\.claude\hooks\log-event.sh lines 46-56 (slice #572): The 'session_id stamping' fix is a tautological no-op: it extracts session_id from the incoming line and merges the same value back into the same line; it can never repair a missing/empty sid.
- [minor] C:\Users\vojte\bin\jq.exe vs winget jq install (bootstrap.sh step 7): A manually-copied, Mark-of-the-Web-flagged duplicate of jq.exe in ~/bin shadows the winget install in Git-Bash-derived PATHs; bootstrap.sh neither created nor detects this shadow, so environment drift between 'works in my shell' and 'fails in hooks' is invisible.
- [minor] settings.json PostToolUse(Edit|MultiEdit|Write) registration: Pipes a plain-text (non-JSON) line into log-event.sh, relying on jq's soft-failure to skip sid extraction — a fragile implicit contract with the JSONL appender (works only because of LOGFILE override + error suppression).

## OPPORTUNITIES
- Remove the jq dependency from the capture path entirely: replace the 7 inline jq pipelines with one small parameterized script (e.g. `bash .claude/hooks/log-tool-event.sh <event_type>`) that reads stdin once and extracts fields in pure bash or via the python3 already required by the dashboard — this also fixes per-registration telemetry naming for free
- Make failure loud: log-event.sh should write an error beacon ({"hook":"log-event","error":"empty-stdin"}) when stdin is empty instead of exiting silently, and the dashboard hooks view should surface error beacons as a red badge — the #644 telemetry then actually detects the failure class that killed capture
- Emit a session_start event from session-start.sh (one line through log-event.sh) and source session_stop's sid from the stdin payload like every other event, giving /api/runs real session boundaries instead of first-tool-event inference
- Eliminate the machine-level hazard: delete or rename C:\Users\vojte\bin\jq.exe (or Unblock-File it) and have bootstrap.sh/session-start.sh assert `jq --version` actually EXECUTES (not just `command -v jq`) in the hook-equivalent environment
- Decide the SSE question per YAGNI: either wire the Live tab to /api/events with an EventSource client (fixing the Last-Event-ID duplicate-replay bug and switching to incremental file tailing), or delete the dead endpoint and keep honest 30 s polling
- Unify root resolution: all hook beacons and dashboard-autostart should use the same git-common-dir resolution log-event.sh already has, so worktree-isolated subagent sessions feed the main dashboard instead of invisible worktree logs
- Quarantine synthetic data permanently: require a marker field (e.g. {"synthetic":true}) on any test-written event, filter it in _serve_runs/measured-edge derivation, and never write fixtures to the live log path
- Change the qa-tester hook-fire verification route from 'invoke the script by hand' to 'run a real `claude -p` session with --debug and assert workflow-events.jsonl gained lines with non-empty session_id/command' — the by-hand route is exactly what masked this outage (the technique is proven: it reproduced and root-caused the failure in this audit)
- Bucket resilience in /api/runs: treat empty-sid events as per-line orphans (or attach to the temporally-nearest session) rather than one merged 'unknown' run, and make the ?session= scan interleave-safe by scanning the full window instead of break-on-non-match
